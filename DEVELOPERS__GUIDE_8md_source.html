16 ## Testing
17 
18  Each repository contains a src folder. Within the src folder exists a
19  library folder: libtools, libcsg etc... and a tools folder. A tests folder
20  should also exist in the src folder. If it does not you should create one.
21 
22  For every new object and algorithm created there should exist a test. We
23  use the Boost libraries testing framework. Good documentation can be found
24  here:
25 
26  - [Boost link](https://www.ibm.com/developerworks/aix/library/au-ctools1_boost/)
27 
28  We will outline the general workflow here using the vec object in
29  votca::tools. This object only has a header file it is in:
30  tools/include/votca/tools/vec.h
31 
32  Determine if a tests folder has already been created or not in /src if it
33  has not take a look at what was done in the votca-tools repo.
34 
35  1. Create a test file in [tools/src/tests/](https://github.com/votca/tools/tree/master/src/tests)test_vec.cc
36  must have the same name as what appears in the foreach in the
37  CMakeLists.txt file. And place the following contents
38 
39 ```
40 #define BOOST_TEST_MAIN
41 
42 #define BOOST_TEST_MODULE vec_test
43 #include <boost/test/unit_test.hpp>
44 #include <exception>
45 
46 #include <votca/tools/vec.h>
47 
48 using namespace std;
49 using namespace votca::tools;
50 
51 BOOST_AUTO_TEST_SUITE(vec_test)
52 
53 
54 BOOST_AUTO_TEST_CASE(test1){
55  vecv;
56  BOOST_CHECK_EQUAL(...);
57  BOOST_CHECK_EQUAL(...);
58  :
59 }
60 BOOST_AUTO_TEST_CASE(test2){
61  vecv;
62  BOOST_CHECK_EQUAL(...);
63  BOOST_CHECK_EQUAL(...);
64  :
65 }
66 :
67 BOOST_AUTO_TEST_SUITE_END()
68 ```
69 
70 Replace the '...' and ':' with the appropriate syntax.
71 For more info on which boost test macros to use refer to the boost documentation
72 
73  2. To compile and test the code create a folder tools/build and run the following
74  commands:
75 
76 ```
77 cmake -DENABLE_TESTING=ON ../
78 make
79 make test
80 ```
81 
82 Ensure you have an up to date version of cmake or use cmake3

84 ## CPP Codeing Style Guide
85 
86  VOTCA uses the clang formatter to automatically ensure consistent style.
87  The style is essentially the same, as what is used by google, with the
88  exception that types and = are aligned to make the code more readable.
89  Have a look at `.clang-format file` in the
90  [main votca repository](https://github.com/votca/votca/blob/master/.clang-format) for details.
91 
92 
93 To run the clang-format function on file.cc
94 
95 ```
96 clang-format -i -style=file file.cc
97 ```
98 
99 '-i' ensures it will make change to file.cc, omitting the '-i' will display the
100  changes without implementing them.
101 '-style=file' ensures the format is read from the .clang-format file otherwise
102  it will use a default style guide.
