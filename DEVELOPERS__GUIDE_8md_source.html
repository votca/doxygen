46 ## CPP Tips
47 
48  Here are a few general tips that should be followed:
49 
50 ### Files ###
51  * each class goes into a separate file
52  * filename is name of class in lowercase
53 
54 ### Header Files ###
55  * One class, one header.
56  * When creating header gruards use the following form, where "VOTCA-REPO-NAME" is replaced by whichever repo the header is in tools/csg/ctp/xtp, and where "CLASS-NAME" is replaced by the name of the class described in the header file:
57 ```
58 #ifndef _VOTCA_VOTCA-REPO-NAME_CLASS-NAME_H
59 #define _VOTCA_VOTCA-REPO-NAME_CLASS-NAME_H
60 :
61 Code
62 :
63 #endif // _VOTCA_VOTCA-REPO-NAME_CLASS-NAME_H
64 ```
65  * Never use the "using namespace" in a header file.
66  * Avoid using includes in header files. If possible forward declare a class instead.
67 
68 ### Braces ###
69  * in functions/classes, the { is in the next line
70  * for for loops, if, ..., the { is n the same line as if,for
71 
72 ### Classes ###
73  * normally begin in upper case
74  * exceptions: classes which define "small" types (e.g. `vec`, `matrix`)
75  * order in class definition:
76  * first `public` all functions
77  * then `private`/`protected` all member variables
78  * then `private`/`protected` member functions
79  * no rule where to define a `public typedef` in the class
80  * all member variables are `private`/`public` (again exception e.g. `vec`, `matrix`)
81  * maximum one-line-function implementation in class declaration, everything else moves to separate file or inline at end of header.
82 
83 ### Naming in Classes ###
84  * all member variables are in lower case and end with `_`
85  * all functions start with upper case, no `_` in names
86  * exception: `get`/`set` functions
87  * for consistency all Ids should start at 0 not 1
88 
89 ### get/set Functions ###
90  * get/set functions start with a lowercase set/get (these are only functions which directly set/get a private member variable)
91  * get must return a constant reference and keep the `class const`: `const int &getId() const;`
92  * set only sets the member, e.g. `void setId(const int &id) {_id=id;}`
93 
94 ### Functions ###
95  * Make functions short.
96  * Functions should not have more than one use. So use boolean arguments sparingly.
97 
98 ### General ###
99  * Do not comment out code, if you do not use it delete it.
100  * Variables should have clear and explicit names.
101  * Do not duplicate code.
102  * Functions should have no more than 3 arguments. Otherwise create a class.
103  * XYZ positions should be described using tools::vec, 3x3 matrices tools::matrix, or classes and or functions in the eigen library.
104  * Readability is more important the elegant design.
105  * Leave the code better than you found it.
106  * Use pointers sparingly and especially try not to pass them around objects. Prefer references.
107  * Do not write code, which you may use in the future. Only write code you will use now. Write code, you need later, later. This avoids cluttering the codebase with unused "at some point we will need this functions".

109 ## Testing
110 
111  Each repository contains a src folder. Within the src folder exists a
112  library folder: libtools, libcsg etc... and a tools folder. A tests folder
113  should also exist in the src folder. If it does not you should create one.
114 
115  For every new object and algorithm created there should exist a test. We
116  use the Boost libraries testing framework. Good documentation can be found
117  here:
118 
119  - [Boost link](https://www.ibm.com/developerworks/aix/library/au-ctools1_boost/)
120 
121  We will outline the general workflow here using the vec object in
122  votca::tools. This object only has a header file it is in:
123  tools/include/votca/tools/vec.h
124 
125  Determine if a tests folder has already been created or not in /src if it
126  has not take a look at what was done in the votca-tools repo.
127 
128  1. Create a test file in [tools/src/tests/](https://github.com/votca/tools/tree/master/src/tests)test_vec.cc
129  must have the same name as what appears in the foreach in the
130  CMakeLists.txt file. And place the following contents
131 
132 ```
133 #define BOOST_TEST_MAIN
134 
135 #define BOOST_TEST_MODULE vec_test
136 #include <boost/test/unit_test.hpp>
137 #include <exception>
138 
139 #include <votca/tools/vec.h>
140 
141 using namespace std;
142 using namespace votca::tools;
143 
144 BOOST_AUTO_TEST_SUITE(vec_test)
145 
146 
147 BOOST_AUTO_TEST_CASE(test1){
148  vecv;
149  BOOST_CHECK_EQUAL(...);
150  BOOST_CHECK_EQUAL(...);
151  :
152 }
153 BOOST_AUTO_TEST_CASE(test2){
154  vecv;
155  BOOST_CHECK_EQUAL(...);
156  BOOST_CHECK_EQUAL(...);
157  :
158 }
159 :
160 BOOST_AUTO_TEST_SUITE_END()
161 ```
162 
163 Replace the '...' and ':' with the appropriate syntax.
164 For more info on which boost test macros to use refer to the boost documentation
165 
166  2. To compile and test the code create a folder tools/build and run the following
167  commands:
168 
169 ```
170 cmake -DENABLE_TESTING=ON ../
171 make
172 make test
173 ```
174 
175 Ensure you have an up to date version of cmake or use cmake3

177 ## CPP Codeing Style Guide
178 
179  VOTCA uses the clang formatter to automatically ensure consistent style.
180  The style follows the google style fomatting rules.
181  Have a look at `.clang-format file` in the
182  [main votca repository](https://github.com/votca/votca/blob/master/.clang-format) for details.
183 
184 
185 To run the clang-format function on file.cc
186 
187 ```
188 clang-format -i -style=file file.cc
189 ```
190 
191 '-i' ensures it will make change to file.cc, omitting the '-i' will display the
192  changes without implementing them.
193 '-style=file' ensures the format is read from the .clang-format file otherwise
194  it will use a default style guide.
195 
196 By default tabs should not be used to indent, avoid inserting '\t', it is
197 preferable that spaces be used instead.

199 ## CPP Comment Guide
200 
201  It is preferential that the following guidelines be followed when adding
202  comments to code:
203 
204  1. The `/* */` comment blocks should be avoided and the `//` used in their
205  place. This is so that the `/* */` comment blocks can be easily used for
206  debugging.
207  2. It would be preferential that the following doxygen commenting stencil be
208  used in the header files above each class and function description.
209 
210 ```
211 /**
212 * \brief function/class summary
213 *
214 * Detailed function/class description if needed
215 *
216 * @param[in] - description of parameter 1
217 * @param[out] - description of parameter 2
218 * @param[in,out] - description of parameter 3
219 * :
220 * @return - description of return type
221 */
222 ```
223 
224 The doxygen commenting will help future developers maintain the code, in its
225 fully compiled state it may be found at: http://doc.votca.org
226 
227 NOTE: Compilation of the doxygen documentation is automated when code is merged
228 into the master votca branch!

230 ## Updating Git Submodules
231 
232  Votca with all of its repos can be build by using the parent [votca repo](https://github.com/votca/votca). All
233  the other necessary repos appear as submodules in the parent repo. It is worth
234  noting that the submodules are not automatically updated whenever changes are
235  made to their respective master branches. In essence a submodule refers to a
236  specific commit of the repo it represents. If a new commit is merged into the
237  master branch of a repository the submodule state in the parent repo has to be
238  updated for the commit to propagate to the parent votca repository.
239 
240  To update the state of a submodule the following commands can be used:
241 
242 ```
243 git submodule foreach git checkout master
244 git submodule foreach git pull
245 git add -u
246 git commit -m "update all submodules"
247 ```
