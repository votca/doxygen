31 // to sum it up: instead of having one "thread" doing all your work (the whole tracetory),
32 // you may split it into single frames and distribute it among many "workers".
33 // a solid choice is: number of cores = number of workers.
34 // you, as the user, are required to define how to initialize and merge your workers.
35 // the main part of the program, EvalConfiguration, is shifted to the Worker class
36 // but other than that stays untouched compared to a non-threaded version

146 // the user is required to define how to merge the single data
147 // belonging to each thread into the main data belonging to CsgTestApp
148 void CsgTestApp::MergeWorker(Worker *worker) {
149  RDFWorker * myRDFWorker;
150 // cast generel Worker into your derived worker class(here RDFWorker)
151  myRDFWorker = dynamic_cast<RDFWorker*> (worker);
152 
153 // the next comment block explains how mutexes are used internally for this function:
154 // mutexes are used to exclusively work on data
155 // e.g., if you read or write global data, make sure that nobody else (i.e. no other worker)
156 // works on that very same piece of data at the same time; otherwise,
157 // you will end up with wrong results that you struggle to understand
158 // the parent class handles a "merging mutex" for you internally; this is what happens:
159 // first, a mutex is created, e.g.
160 // Mutex rdfMutex;
161 // then, for each worker, the mutex is first locked
162 // rdfMutex.Lock())
163 // and MergeWorker(worker) is called (i.e. the code you define here is executed)
164 // after MergeWorker exits, the mutex is unlocked
165 // rdfMutex.Unlock();
166 // and allows other threads to get a lock and start merging
167 
168 // now follows your code
169 
170 // merging of data in this simple example is easy and does not have to follow
171 // the original order of frames (since plain summing is commutative)
172  _rdf.data().y() = _rdf.data().y() + myRDFWorker->_rdf.data().y();
173 
174 }
